# 项目介绍 / Project Introduction

## 结合AI的自动化Web漏洞发掘：融合LLM与传统扫描器的混合框架

## AI-Enhanced Automated Web Vulnerability Discovery: A Hybrid Framework Integrating LLMs with Traditional Scanners

---

## 一、研究背景 / Research Background

### 中文

Web 应用漏洞是当前网络安全中最主要的攻击面之一。根据 OWASP Top 10 (2021)，注入攻击、访问控制缺陷、跨站脚本等问题持续占据威胁榜首。在安全应急响应中心（SRC）的日常运营中，安全工程师需要处理海量的漏洞报告，其中大量来自自动化扫描工具的输出。这些工具虽然覆盖广泛，但产生的高误报率迫使安全团队投入大量人力进行人工复核，成为制约自动化安全测试效率的核心瓶颈。

大语言模型（LLM）——如 GPT-4、Claude 等——在 2023-2025 年间展现出强大的语义理解与逻辑推理能力。不同于传统的规则匹配引擎，LLM 能够理解 HTTP 请求/响应的上下文语义、识别异常模式、并给出解释性判断。这为自动化漏洞检测中的误报过滤提供了一种全新的技术路径：让 LLM 充当"安全专家"，在扫描器输出与人工确认之间增加一层智能化分析，以机器速度完成类人的判断。

### English

Web application vulnerabilities remain one of the most significant attack surfaces in cybersecurity. According to the OWASP Top 10 (2021), injection attacks, broken access control, and cross-site scripting continue to dominate the threat landscape. In the daily operations of Security Response Centres (SRCs), security engineers must triage an overwhelming volume of vulnerability reports, many of which originate from automated scanning tools. While these tools provide broad coverage, their high false-positive rates force security teams to invest substantial human effort in manual verification — a bottleneck that fundamentally limits the efficiency of automated security testing.

Large Language Models (LLMs) — such as GPT-4 and Claude — have demonstrated remarkable semantic understanding and logical reasoning capabilities between 2023 and 2025. Unlike traditional rule-matching engines, LLMs can comprehend the contextual semantics of HTTP request/response pairs, recognise anomalous patterns, and deliver explanatory judgments. This presents a novel technical pathway for false-positive filtering in automated vulnerability detection: positioning an LLM as a "security expert" that adds an intelligent analysis layer between scanner output and human confirmation, delivering human-like judgment at machine speed.

## 二、领域目前不足 / Current Limitations in the Field

### 中文

**1. 传统扫描器误报率高**

Amankwah et al. (2020) 在 *Software: Practice and Experience* 期刊上对 8 款主流 Web 漏洞扫描器（含商业和开源）进行了实证评估。在 DVWA 靶机测试中，OWASP ZAP —— 最广泛使用的开源扫描器之一 —— 精确率仅为 **0.56**，意味着 44% 的告警是误报；Arachni 和 Vega 的精确率同为 0.56。多个扫描器在 OWASP WBE 分类体系中被归为"everything is vulnerable（什么都是漏洞）"类别。Doupé et al. (2010) 在 DIMVA 会议上对 11 款黑盒扫描器的分析同样指出，不同扫描器的检测能力差异悬殊，且没有一款能在保持高召回率的同时控制误报。

**2. 缺乏上下文感知能力**

传统扫描器依赖固定签名和预定义 Payload 字典。当目标应用的响应中包含与漏洞特征相似的文本时（如页面本身的错误提示被误判为数据库报错），扫描器无法区分——它没有"理解"能力。这类误报在反射型 XSS 和 SQL 注入检测中尤为常见。

**3. SRC 运营中的实际痛点**

在 SRC 场景下，安全运营人员每天面对数百条自动化告警。每一条误报都意味着 5-15 分钟的浪费；累积起来，误报造成的"告警疲劳"会降低整个安全团队的响应效率，甚至导致真实漏洞被忽略。

**4. LLM 在安全领域的应用仍属早期**

Happe & Cito (2023) 在 ESEC/FSE 上首次探索了用 LLM 辅助渗透测试；Deng et al. (2024) 在 USENIX Security 上提出了 PentestGPT 多模块框架。然而，**尚无已发表的系统性研究将 LLM 专门作为扫描后误报过滤机制，并在可控实验中量化其效果**——这是本研究要填补的空白。

### English

**1. High False-Positive Rates in Traditional Scanners**

Amankwah et al. (2020), in *Software: Practice and Experience*, conducted an empirical evaluation of eight mainstream web vulnerability scanners (both commercial and open-source) against DVWA and WebGoat. OWASP ZAP — one of the most widely adopted open-source scanners — achieved a precision of only **0.56**, meaning 44% of its alerts were false positives. Arachni and Vega recorded identical precision scores. Multiple scanners fell into the "everything is vulnerable" classification under the OWASP Web Benchmark Evaluation framework. Earlier, Doupé et al. (2010) at DIMVA analysed 11 black-box scanners and reached a similar conclusion: detection capability varies dramatically across tools, and no single scanner simultaneously achieves high recall and controlled false-positive rates.

**2. Lack of Contextual Awareness**

Traditional scanners rely on fixed signatures and predefined payload dictionaries. When an application response contains text that superficially resembles a vulnerability indicator — for instance, when a page's built-in error message is misclassified as a database error — the scanner cannot differentiate. It has no capacity for semantic "understanding." This class of false positives is particularly prevalent in reflected XSS and SQL injection detection.

**3. Operational Pain Points in SRC Workflows**

In SRC environments, security operations personnel face hundreds of automated alerts daily. Each false positive represents 5–15 minutes of wasted triage time. Cumulatively, the "alert fatigue" induced by false positives degrades the responsiveness of the entire security team and can cause genuine vulnerabilities to be overlooked.

**4. LLM Applications in Security Remain Nascent**

Happe & Cito (2023) at ESEC/FSE first explored LLM-assisted penetration testing; Deng et al. (2024) at USENIX Security proposed the PentestGPT multi-module framework. However, **no published study has systematically evaluated LLMs specifically as a post-scanning false-positive filter with quantitative metrics in controlled experiments** — the gap this research addresses.

## 三、技术方案 / Technical Approach

### 中文

**总体架构：三层混合系统**

```
┌────────────────────────────────────────────┐
│  表现层：React Web Dashboard                │
│  扫描管理 · 漏洞展示 · 统计分析 · 报告导出    │
├────────────────────────────────────────────┤
│  分析层：LLM 验证引擎                        │
│  Prompt 构造 · 多模型支持 · 结果解析          │
│  三种策略：Zero-shot / Few-shot / CoT       │
├────────────────────────────────────────────┤
│  执行层：扫描器编排引擎                       │
│  Nuclei (通用CVE/配置) · sqlmap (SQL注入)    │
│  dalfox (XSS) · ffuf (目录爆破)             │
│  Docker 沙箱隔离 · 结果结构化收集              │
└────────────────────────────────────────────┘
```

**技术栈**

| 层 | 技术选型 |
|---|---------|
| 后端 | Python 3.12 + FastAPI + Celery（异步任务队列） |
| 数据库 | PostgreSQL + SQLAlchemy ORM |
| 前端 | React 18 + TypeScript + Ant Design |
| LLM 接入 | Anthropic API (Claude) / OpenAI API (GPT-4)，可插拔架构 |
| 扫描引擎 | Nuclei, sqlmap, dalfox, ffuf（通过 subprocess 调用） |
| 容器化 | Docker + Docker Compose（系统部署 + 靶机环境） |

**LLM 提示策略**

系统将采用三种 Prompt 策略进行对比实验：

- **Zero-shot（零样本）：** 直接给定 HTTP 请求/响应，要求判断真伪漏洞
- **Few-shot（少样本）：** 在 Prompt 中提供 3 组经过标注的 TP/FP 示例
- **Chain-of-Thought（思维链）：** 要求 LLM 先逐步推理，再给出最终判断

LLM 输出格式为结构化 JSON：`{verdict, confidence, reasoning, severity_reassessment}`

**实验设计**

| 要素 | 说明 |
|------|------|
| 测试靶机 | OWASP Juice Shop, DVWA, WebGoat, bWAPP, VulnHub Custom（共5个） |
| 基线 | 纯扫描器输出（无 LLM） |
| 实验条件 | Scanner + LLM Zero-shot / Few-shot / CoT |
| 人工基线 | 2 名安全研究方向人员独立标注 |
| 评估指标 | Precision, Recall, F1-Score, False Positive Rate, Specificity |
| 统计检验 | McNemar's test（配对名义数据显著性检验） |

### English

**Overall Architecture: Three-Tier Hybrid System**

```
┌────────────────────────────────────────────┐
│  Presentation Layer: React Web Dashboard   │
│  Scan management · Findings display        │
│  Statistical analytics · Report export     │
├────────────────────────────────────────────┤
│  Analysis Layer: LLM Verification Engine   │
│  Prompt construction · Multi-model support │
│  Verdict parsing                           │
│  Three strategies: Zero-shot/Few-shot/CoT  │
├────────────────────────────────────────────┤
│  Execution Layer: Scanner Orchestration    │
│  Nuclei (CVE/config) · sqlmap (SQLi)       │
│  dalfox (XSS) · ffuf (dir brute-force)     │
│  Docker sandboxing · Structured collection │
└────────────────────────────────────────────┘
```

**Technology Stack**

| Tier | Technology Choices |
|------|-------------------|
| Backend | Python 3.12 + FastAPI + Celery (async task queue) |
| Database | PostgreSQL + SQLAlchemy ORM |
| Frontend | React 18 + TypeScript + Ant Design |
| LLM Integration | Anthropic API (Claude) / OpenAI API (GPT-4), pluggable architecture |
| Scan Engines | Nuclei, sqlmap, dalfox, ffuf (invoked via subprocess) |
| Containerisation | Docker + Docker Compose (system deployment + target environments) |

**LLM Prompting Strategy**

Three prompting strategies will be compared experimentally:

- **Zero-shot:** Directly provide HTTP request/response and request a TP/FP verdict
- **Few-shot:** Include 3 annotated TP/FP examples within the prompt
- **Chain-of-Thought (CoT):** Instruct the LLM to reason step-by-step before declaring a verdict

LLM output is structured as JSON: `{verdict, confidence, reasoning, severity_reassessment}`

**Experimental Design**

| Element | Description |
|---------|-------------|
| Test Targets | OWASP Juice Shop, DVWA, WebGoat, bWAPP, VulnHub Custom (5 total) |
| Baseline | Scanner-only output (no LLM) |
| Experimental Conditions | Scanner + LLM Zero-shot / Few-shot / CoT |
| Human Baseline | 2 independent reviewers with security research background |
| Metrics | Precision, Recall, F1-Score, False Positive Rate, Specificity |
| Statistical Test | McNemar's test for paired nominal data |

## 五、如何实现项目 / How to Implement the Project

### 中文

**阶段一：环境与基础设施（第 1-3 周）**

1. 安装 Python 3.12，初始化 FastAPI 项目结构
2. 配置 PostgreSQL 数据库，使用 SQLAlchemy 建立数据模型（ScanTask, Finding, LLMAnalysis）
3. 使用 Docker Compose 拉取并部署 5 个靶机（Juice Shop, DVWA, WebGoat, bWAPP, VulnHub）
4. 安装扫描工具：nuclei, sqlmap, dalfox, ffuf，验证各工具命令行可用
5. 注册 LLM API（Anthropic / OpenAI），编写 API 调用测试脚本

**阶段二：扫描器编排引擎（第 4-6 周）**

1. 编写 ScannerOrchestrator 模块，通过 Python subprocess 调用各扫描器
2. 实现扫描结果解析器，将不同工具的输出统一为结构化格式（JSON）
3. 实现异步任务队列（Celery），支持扫描任务的排队、执行、状态跟踪
4. 编写 Docker 沙箱模块，确保每次扫描在隔离网络环境中执行

**阶段三：LLM 分析引擎（第 7-9 周）**

1. 设计 Prompt 模板（System Prompt + Context + Output Schema）
2. 实现三种提示策略：Zero-shot、Few-shot（3 组标注示例）、Chain-of-Thought
3. 编写 LLMClient 抽象层，支持 Anthropic 和 OpenAI 双后端切换
4. 实现结构化输出解析（JSON → Verdict 对象），含异常处理
5. 编写 Prompt 策略对比实验脚本

**阶段四：前端 Dashboard（第 10-13 周）**

1. 使用 React 18 + TypeScript + Ant Design 搭建前端项目
2. 实现页面：扫描任务创建、任务列表与状态、漏洞详情（含 LLM 分析结论）、统计图表
3. 实现用户设置页（API Key 配置、模型选择）
4. 与后端 API 联调，完成端到端数据流

**阶段五：实验与评估（第 14-17 周）**

1. 为 5 个靶机构建 Ground Truth（通过人工代码审查 + 官方文档标注真实漏洞列表）
2. 运行基线实验：纯扫描器扫描 → 记录所有发现
3. 运行 LLM 验证实验：每种 Prompt 策略处理所有扫描发现
4. 人工标注：2 名评审人独立标注同一批发现，计算 Cohen's Kappa
5. 计算并对比各组 Precision / Recall / F1 / FPR / Specificity
6. 执行 McNemar's test 检验差异显著性
7. 按漏洞类型（SQLi, XSS, SSRF 等）做分层分析

**阶段六：论文撰写（第 18-24 周，与实验阶段重叠）**

1. 引言与文献综述（基于前期阅读整理）
2. 方法论章节
3. 系统设计与实现章节
4. 实验结果与分析章节
5. 讨论、局限性与未来工作
6. 全文修订与格式检查

### English

**Phase 1: Environment and Infrastructure (Weeks 1–3)**

1. Set up Python 3.12 and initialise the FastAPI project structure
2. Configure PostgreSQL; define data models (ScanTask, Finding, LLMAnalysis) using SQLAlchemy
3. Use Docker Compose to pull and deploy five target applications (Juice Shop, DVWA, WebGoat, bWAPP, VulnHub)
4. Install scanning tools — nuclei, sqlmap, dalfox, ffuf — and verify CLI functionality
5. Register LLM API access (Anthropic / OpenAI) and write API connectivity test scripts

**Phase 2: Scanner Orchestration Engine (Weeks 4–6)**

1. Develop the ScannerOrchestrator module, invoking each scanner via Python subprocess
2. Implement result parsers that normalise heterogeneous tool outputs into a unified JSON schema
3. Implement an asynchronous task queue (Celery) for scan queuing, execution, and status tracking
4. Develop a Docker sandboxing module to ensure scans run in isolated network environments

**Phase 3: LLM Analysis Engine (Weeks 7–9)**

1. Design prompt templates (System Prompt + Context + Output Schema)
2. Implement three prompting strategies: Zero-shot, Few-shot (3 annotated examples), Chain-of-Thought
3. Develop an LLMClient abstraction layer supporting Anthropic and OpenAI backends with hot-switching
4. Implement structured output parsing (JSON → Verdict object) with exception handling
5. Write comparison experiment scripts for prompting strategy evaluation

**Phase 4: Frontend Dashboard (Weeks 10–13)**

1. Scaffold the frontend project using React 18 + TypeScript + Ant Design
2. Implement pages: scan creation, task list with status, vulnerability details (with LLM analysis conclusions), statistical charts
3. Implement a user settings page (API Key configuration, model selection)
4. Integrate with the backend API for end-to-end data flow

**Phase 5: Experimentation and Evaluation (Weeks 14–17)**

1. Construct ground truth for all five targets through manual code review and official documentation
2. Run baseline experiments: scanner-only → record all findings
3. Run LLM verification experiments: each prompting strategy processes all scanner findings
4. Human annotation: two independent reviewers classify the same finding set; compute Cohen's Kappa
5. Calculate and compare Precision / Recall / F1 / FPR / Specificity across all conditions
6. Perform McNemar's test for statistical significance
7. Conduct stratified analysis by vulnerability type (SQLi, XSS, SSRF, etc.)

**Phase 6: Dissertation Writing (Weeks 18–24, overlapping with experimentation)**

1. Introduction and literature review
2. Methodology chapter
3. System design and implementation chapter
4. Experimental results and analysis chapter
5. Discussion, limitations, and future work
6. Full-text revision and formatting
